    조직/팀 프로필 (OrgProfile / TeamProfile)

    TL;DR

    • Organization: 회사(본사) 단일 레코드. Admin만 편집 가능.
    • Team: 센터 정보. name(센터명)은 생성 후 변경 불가(Immutable).
    • Phase 1은 비활성화만 허용, Hard delete 금지.

    🏢 1. Organization (회사) 기본 필드

    필드 타입 설명 비고
    id UUID 레코드 식별자 시스템 자동 생성
    legal_name string 법인명 (사업자 등록 기준) 필수
    biz_reg_no string 사업자등록번호 필수
    address string 사업장 주소 필수
    phone string 대표 연락처 선택
    email string 대표 이메일 선택
    created_at datetime 최초 생성 일시 시스템 자동
    updated_at datetime 최종 수정 일시 시스템 자동

    Phase 1: Organization은 단일 레코드(1개). Admin이 생성·편집. PDF 자동삽입 등 서식 자동화는 Phase 2.

    🏪 2. Team (센터) 기본 필드

    필드 타입 설명 비고
    id UUID 레코드 식별자 시스템 자동 생성
    name string 센터명 (Immutable) 생성 후 변경 불가
    address string 센터 주소 선택 (수정 가능)
    contact_phone string 센터 연락처 선택 (수정 가능)
    manager_name string 담당자 이름 선택 (수정 가능)
    status enum ACTIVE / INACTIVE 비활성화만 허용
    deactivated_at datetime 비활성화 일시 INACTIVE 시 자동
    deactivated_reason string 비활성화 사유 선택 입력
    org_id UUID 소속 Organization 외래키
    created_at datetime 생성 일시 시스템 자동
    updated_at datetime 수정 일시 시스템 자동

    ⚠️ Team.name Immutable 정책

    • 팀명(센터명)은 최초 생성 시에만 입력 가능합니다.
    • 생성 이후 PATCH 요청에서 name 필드는 무시됩니다 (서버가 변경 거부).
    • 관련: RBAC 권장 정책값, API 최소세트

    ⚠️ Team Hard Delete 금지

    • Phase 1에서 Team 삭제는 금지됩니다.
    • 비활성화(PATCH /admin/teams/{id}/deactivate)만 허용.
    • Inactive 팀은 WorkOrder 신규 배정이 차단됩니다.

    👤 3. User Lifecycle

    상태 설명 영향
    ACTIVE 정상 활성 상태 로그인/배정 가능
    INACTIVE 비활성화 상태 로그인 차단, 신규 배정 차단

    Hard delete 금지: 데이터 무결성을 위해 User 레코드는 물리 삭제하지 않습니다. 비활성화(deactivated_at + deactivated_reason 기록)만 허용.

    🔐 4. 편집 권한

    대상 Admin Team Manager Technician
    Organization 전체 ✅ ❌ ❌
    Team 전체 (name 제외) ✅ ⚠️ (자기 팀만) ❌
    Team 주소/연락처/담당자 ✅ ✅ (자기 팀만) ❌
    Team.name ❌ (불가) ❌ (불가) ❌ (불가)
    Team 비활성화 ✅ ❌ ❌
    User 비활성화 (TECH) ✅ ✅ (자기 팀만) ❌
